# @Configuration Best Practices

## 1. What is @Configuration?

`@Configuration` is a class-level annotation that indicates a class declares one or more `@Bean` methods. These classes are processed by the Spring container to generate bean definitions.

## 2. @Configuration vs @Component

| Aspect         | @Configuration                        | @Component                |
| -------------- | ------------------------------------- | ------------------------- |
| Purpose        | Define beans explicitly               | General-purpose component |
| Proxy behavior | CGLIB proxies (by default)            | No proxy                  |
| Use case       | Complex bean wiring, third-party libs | Simple components         |

```java
@Configuration
public class AppConfig {
    @Bean
    public MyService myService() {
        return new MyService(); // Proxied for bean internal calls
    }
}
```

## 3. Best Practices

### 3.1 Use @Configuration for Bean Wiring

```java
@Configuration
public class WebClientConfig {
    @Bean
    public WebClient webClient(WebClientBuilder builder) {
        return builder
            .baseUrl("https://api.example.com")
            .defaultHeader("Authorization", "Bearer token")
            .build();
    }
}
```

### 3.2 Avoid @Configuration for Simple Components

```java
// Good: Use @Component for simple services
@Service
public class UserService { }

// Bad: Overengineering with @Configuration
@Configuration
public class UserServiceConfig {
    @Bean
    public UserService userService() {
        return new UserService();
    }
}
```

### 3.3 Inject Dependencies via Method Parameters

```java
@Configuration
public class ServiceConfig {
    @Bean
    public OrderService orderService(ProductRepository productRepo, CustomerRepository customerRepo) {
        return new OrderServiceImpl(productRepo, customerRepo);
    }
}
```

### 3.4 Use @Profile for Environment-Specific Beans

```java
@Configuration
@Profile("dev")
public class DevConfig {
    @Bean
    public String databaseUrl() {
        return "jdbc:h2:mem:devdb";
    }
}

@Configuration
@Profile("prod")
public class ProdConfig {
    @Bean
    public String databaseUrl() {
        return "jdbc:postgresql://prod-server:5432/db";
    }
}
```

### 3.5 Use @Conditional for Fine-Grained Control

```java
@Configuration
@ConditionalOnProperty(name = "feature.enabled", havingValue = "true")
public class FeatureConfig {
    @Bean
    public FeatureService featureService() {
        return new FeatureService();
    }
}
```

## 4. Common Mistakes

### 4.1 Calling @Bean Methods Within @Configuration Class

```java
@Configuration
public class BadConfig {
    @Bean
    public ServiceA serviceA() {
        return new ServiceA();
    }

    @Bean
    public ServiceB serviceB() {
        // BAD: Direct method call bypasses proxy
        return new ServiceB(serviceA());
    }
}
```

**Solution:** Use dependency injection instead:

```java
@Configuration
public class GoodConfig {
    @Bean
    public ServiceA serviceA() {
        return new ServiceA();
    }

    @Bean
    public ServiceB serviceB(ServiceA serviceA) {
        return new ServiceB(serviceA);
    }
}
```

### 4.2 Overusing @Configuration

Don't use `@Configuration` for every class. Use `@Component`, `@Service`, `@Repository` for application components.

## 5. External Configuration with @ConfigurationProperties

```java
@ConfigurationProperties(prefix = "app")
public record AppProperties(
    String name,
    int maxItems,
    boolean enabled
) {}
```

```yaml
app:
  name: "CRUD Application"
  max-items: 100
  enabled: true
```

```java
@EnableConfigurationProperties(AppProperties.class)
@Configuration
public class AppConfig {
    @Bean
    public AppProperties appProperties() {
        return new AppProperties();
    }
}
```

## 6. Common Patterns

### 6.1 Import Other Configurations

```java
@Configuration
@Import(DatabaseConfig.class)
public class AppConfig { }
```

### 6.2 Conditional Bean Loading

| Annotation                  | Purpose                      |
| --------------------------- | ---------------------------- |
| `@ConditionalOnClass`       | Bean if class exists         |
| `@ConditionalOnMissingBean` | Bean if not already defined  |
| `@ConditionalOnProperty`    | Bean if property matches     |
| `@ConditionalOnExpression`  | Bean if SpEL expression true |

### 6.3 Factory Pattern with @Configuration

```java
@Configuration
public class PaymentConfig {
    @Bean
    public PaymentFactory paymentFactory(Map<String, PaymentProcessor> processors) {
        return new PaymentFactory(processors);
    }
}
```

## 7. Testing with @Configuration

```java
@TestConfiguration
public class TestConfig {
    @Bean
    public MockService mockService() {
        return new MockService("test-data");
    }
}

@SpringBootTest
class MyTest {
    @Autowired
    @Import(TestConfig.class)
    void testWithMock() { }
}
```

## 8. Summary

- Use `@Configuration` for explicit bean wiring and third-party integrations
- Avoid for simple application components (use `@Component`)
- Never call `@Bean` methods directly within a `@Configuration` class
- Leverage `@Profile`, `@Conditional` for environment-specific beans
- Combine with `@ConfigurationProperties` for type-safe external config
