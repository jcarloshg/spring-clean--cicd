### Hibernate Best Practices

1. Use Session management properly:
   - Prefer `getCurrentSession()` for transaction-scoped sessions
   - Always close sessions properly (try-with-resources or finally blocks)

2. Use LAZY loading wisely:
   - Use `@OneToMany(fetch = FetchType.LAZY)` to avoid loading entire object graphs
   - Use `@Fetch(FetchMode.SUBSELECT)` to avoid N+1 queries
   - Initialize lazy collections with `Hibernate.initialize()` or fetch joins

3. Avoid N+1 query problems:
   - Use `JOIN FETCH` in JPQL/HQL for eager loading related entities
   - Use `@BatchSize(size = N)` for batch loading lazy collections
   - Monitor SQL logs during development

4. Use proper transaction handling:
   - Keep transactions short (avoid long-running operations)
   - Use appropriate isolation levels
   - Never commit/rollback in data access layer; delegate to service layer

5. PreferCriteria API or JPQL over native SQL when possible:
   - Better portability across databases
   - Type safety with Criteria API
   - Named queries for reusable statements

6. Handle exceptions properly:
   - Do NOT catch Hibernate exceptions and continue using the session
   - `HibernateException` marks the session as invalid
   - Use `@Repository` with proper exception translation via `@RepositoryAttributes`

7. Use proper caching strategies:
   - Enable L2 cache for frequently accessed, rarely modified data
   - Use `@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)`
   - Monitor cache hit/miss ratios

8. Optimize batch operations:
   - Use `session.setBatchSize(N)` for batch inserts/updates
   - Use `Hibernate.jdbc.batch_size` configuration
   - Clear/persist in batches to avoid memory issues

9. Use proper fetch strategies:
   - Use `FetchType.LAZY` by default
   - Only use `EAGER` for truly required associations
   - Use named entity graphs for specific fetch requirements

10. Always use meaningful indexes:
    - Add indexes on foreign key columns
    - Use `@Index` annotation for composite indexes
    - Analyze query execution plans for large tables

Mastering Hibernate means moving beyond simply getting data in and out of a database, and focusing on system stability, memory management, and architectural boundaries.

Here is a comprehensive list of best practices, followed by advanced tips and tricks for using Hibernate at an enterprise level.

### 1. Architectural & Modeling Best Practices

- **Isolate Persistence from Core Business Logic:** When applying architectural patterns where the business domain is the core, do not let Hibernate annotations (`@Entity`, `@Table`) leak into your pure domain models. Maintain separate persistence entities (mapped to the database) and domain aggregates (containing pure business rules), using repositories to map between them.
- **Implement `equals()` and `hashCode()` Correctly:** Never use the database-generated `@Id` for these methods, as the ID is `null` before the entity is saved, which breaks `Set` collections. Use a unique business key (or a UUID generated at instantiation) to evaluate equality.
- **Favor Composition over Inheritance:** Hibernate supports inheritance strategies (`SINGLE_TABLE`, `JOINED`, `TABLE_PER_CLASS`), but they often lead to complex, slow queries or bloated tables. Use `@Embeddable` and `@Embedded` to group related fields logically without complicating the relational schema.

### 2. Performance & Querying Best Practices

- **Enforce Lazy Loading Exclusively:** Keep `@OneToMany` and `@ManyToMany` as `FetchType.LAZY` (the default). Explicitly change `@ManyToOne` and `@OneToOne` to `LAZY`, as they default to `EAGER`. Eager fetching is the number one cause of memory exhaustion.
- **Use DTO Projections for Read Operations:** If a feature only displays data (e.g., populating a UI table), do not fetch managed entities. Use JPQL, Criteria API, or native queries to fetch data directly into Data Transfer Objects (DTOs). This completely bypasses the Hibernate first-level cache (Persistence Context), saving massive amounts of memory and CPU.
- **Eradicate the N+1 Select Problem:** When you _do_ need to load an entity and its associated lazy collections for modification, do not let Hibernate trigger separate queries in a loop. Use `JOIN FETCH` in your JPQL or leverage `@EntityGraph` to pull everything in a single optimized query.

### 3. Transaction & Session Best Practices

- **Keep Transactions Strictly Scoped:** Transactions should only wrap database operations. Never place HTTP calls, file I/O, or heavy computations inside an `@Transactional` block. This holds database connections hostage and severely limits throughput.
- **Enable JDBC Batching:** Configure `hibernate.jdbc.batch_size` (typically 20-50), `hibernate.order_inserts`, and `hibernate.order_updates`. This allows Hibernate to group multiple statements into a single network packet, drastically reducing latency during bulk operations.
- **Manage the Persistence Context:** If you must update thousands of records in a single transaction using standard entities, the Session will eventually throw an `OutOfMemoryError`. You must periodically call `entityManager.flush()` and `entityManager.clear()` (e.g., every 50 records) to empty the cache.

### 🔥 Advanced Tips & Tricks

- **The `StatelessSession` Trick:** For massive batch inserts or updates where you don't need caching, dirty checking, or lazy loading, use Hibernate's `StatelessSession`. It bypasses the first-level cache entirely, operating almost as fast as raw JDBC.
- **Use `@DynamicUpdate` for Wide Tables:** By default, if you update one column, Hibernate updates _all_ columns in the SQL `UPDATE` statement. If you have a table with 50+ columns or large `TEXT`/`BLOB` fields, add `@DynamicUpdate` to the entity. Hibernate will generate an `UPDATE` statement containing only the modified columns.
- **Leverage `@NaturalId`:** If your entity has a natural business key (like a CURP, email, or ISBN), annotate it with `@NaturalId`. Hibernate optimizes lookups for these fields and manages them in a dedicated region of the second-level cache, making queries by these keys incredibly fast.
- **Log Bound Parameters (Not Just SQL):** Seeing `WHERE id = ?` in the console is unhelpful for debugging. To see the actual values being injected into the query, set the logging level of `org.hibernate.type.descriptor.sql.BasicBinder` to `TRACE`.
- **Hibernate Envers for Easy Auditing:** If you need to keep a historical track of entity changes (who changed what and when), don't write custom triggers. Add the Hibernate Envers dependency and simply annotate your entity with `@Audited`. It automatically creates and manages history tables.
